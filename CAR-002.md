\# Change Assessment Record (CAR-002)



Status



Draft



Title



Verification Class Separation



Purpose



Document the architectural decision to distinguish repository-assisted inspection from byte-faithful cryptographic verification.



Background



A repository-assisted consistency check performed through the GitHub connector indicated SHA256 inconsistencies within the bootstrap-critical artifact set.



Independent verification performed on the workstation using the repository verification procedure successfully verified every bootstrap-critical artifact.



This apparent contradiction required investigation before integrity conclusions could be drawn.



Observation



Evidence was collected from two independent verification paths.



Repository-Assisted Inspection



Observed:



\- repository accessible;

\- bootstrap-critical artifacts present;

\- manifest present;

\- documents readable;

\- connector inspection produced hash inconsistencies.



Additional observation:



The connector representation appeared to contain presentation transformations, including escaped markdown characters.



Workstation Verification



The repository verification procedure:



&#x20;   powershell -ExecutionPolicy Bypass -File .\\verify-bootstrap.ps1



reported:



\- all bootstrap-critical artifacts verified;

\- manifest consistency confirmed;

\- SHA256 verification passed.



No repository corruption was observed.



Analysis



The observed discrepancy was determined to arise from differing verification paths rather than demonstrated repository corruption.



The connector inspection path appears suitable for repository inspection and orientation.



The workstation verification path performs byte-faithful cryptographic verification.



The investigation demonstrated that inspection and cryptographic verification are distinct verification classes.



Findings



Repository-assisted inspection can establish:



\- repository accessibility;

\- repository identity;

\- artifact presence;

\- structural consistency;

\- document readability.



Repository-assisted inspection alone does not establish:



\- byte-for-byte integrity;

\- cryptographic hash equivalence.



Byte-faithful verification establishes:



\- exact file content integrity;

\- cryptographic SHA256 consistency.



Decision



Adopt explicit verification classes within CRH.



Create:



\- BYTE-FAITHFUL-VERIFICATION.md



Update:



\- OBSERVABLE-VERIFICATION.md



Require verification reports to disclose:



\- verification method;

\- evidence inspected;

\- conclusions supported;

\- remaining unknowns;

\- verification limitations.



Operational Rule



Repository-assisted inspection shall not be treated as cryptographic verification unless byte fidelity has been independently demonstrated.



When inspection and byte-faithful verification disagree, the byte-faithful verification result shall be considered authoritative until additional evidence is obtained.



Impact



Positive



\- reduces false integrity conclusions;

\- strengthens Inspection Before Inference;

\- improves verification transparency;

\- establishes an evidence hierarchy;

\- clarifies connector limitations;

\- improves future conformance testing.



Negative



None identified.



Relationship to CRH



This assessment strengthens the Observable Verification model.



It does not alter bootstrap behavior.



It refines how integrity conclusions are classified and disclosed.



Evidence Summary



Observed:



\- connector inspection;

\- workstation verification;

\- successful repository verification;

\- successful Git commit;

\- successful Git push.



Conclusion



The investigation did not demonstrate repository corruption.



The investigation demonstrated that different verification methods possess different evidentiary authority.



CRH now explicitly distinguishes repository-assisted inspection from byte-faithful cryptographic verification.



Next Safe Move



Continue using:



\- repository-assisted inspection for orientation;

\- byte-faithful verification for bootstrap-critical integrity.



Future verification reports should explicitly disclose the verification class used before drawing integrity conclusions.
